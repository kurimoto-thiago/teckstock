1
VPC e Rede
VPC · 4 Subnets · IGW · NAT Gateway · Route Tables
▼
🏗️
Por que este passo aqui?
A VPC é o alicerce de toda a infraestrutura. Nenhum outro recurso AWS pode ser criado sem ela — EC2, RDS, ALB e NAT Gateway todos precisam de subnets que só existem dentro de uma VPC. Criamos a rede primeiro porque tudo o que virá a seguir vive dentro dela. As subnets públicas hospedarão o ALB e o NAT Gateway; as subnets privadas hospedarão o Backend, o Banco e o Monitoring. O NAT Gateway é criado agora porque as instâncias privadas precisarão de acesso à internet para baixar pacotes (npm install, dnf install) durante a configuração.
📥 Precisa: Nada — este é o primeiro passo📤 Fornece: VPC ID📤 Fornece: Subnet IDs (pública e privada)📤 Fornece: NAT Gateway para acesso externo das instâncias privadas📤 Fornece: Variáveis: $VPC_ID, $PUB_A, $PUB_B, $PRIV_A, $PRIV_B
Passo 1 — Criar VPC e obter o ID
# ── Variáveis que usaremos nos próximos passos ──────────────────────────────
APP="techstock"
REGION="us-east-1"

# ── VPC ─────────────────────────────────────────────────────────────────────
# EnableDnsHostnames=true: CRÍTICO — sem isso, o endpoint do RDS não resolve
# EnableDnsSupport=true: ativa o resolver DNS interno da AWS
VPC_ID=$(aws ec2 create-vpc \
  --cidr-block 10.0.0.0/16 \
  --query 'Vpc.VpcId' --output text)

aws ec2 modify-vpc-attribute --vpc-id $VPC_ID --enable-dns-hostnames
aws ec2 modify-vpc-attribute --vpc-id $VPC_ID --enable-dns-support
aws ec2 create-tags --resources $VPC_ID --tags Key=Name,Value=$APP-vpc

echo "VPC_ID=$VPC_ID"  # ANOTE este valor!
Passo 2 — Criar as 4 Subnets (2 públicas + 2 privadas)
# Por que 2 AZs? ALB exige mínimo 2 subnets em AZs diferentes.
# RDS Subnet Group também exige 2 AZs.
# Subnets públicas: hospedam o ALB (precisa de IP público) e o NAT GW
# Subnets privadas: hospedam EC2 Backend, RDS, EC2 Monitoring (sem IP público)

PUB_A=$(aws ec2 create-subnet \
  --vpc-id $VPC_ID --cidr-block 10.0.1.0/24 \
  --availability-zone us-east-1a \
  --query 'Subnet.SubnetId' --output text)

PUB_B=$(aws ec2 create-subnet \
  --vpc-id $VPC_ID --cidr-block 10.0.2.0/24 \
  --availability-zone us-east-1b \
  --query 'Subnet.SubnetId' --output text)

PRIV_A=$(aws ec2 create-subnet \
  --vpc-id $VPC_ID --cidr-block 10.0.10.0/24 \
  --availability-zone us-east-1a \
  --query 'Subnet.SubnetId' --output text)

PRIV_B=$(aws ec2 create-subnet \
  --vpc-id $VPC_ID --cidr-block 10.0.11.0/24 \
  --availability-zone us-east-1b \
  --query 'Subnet.SubnetId' --output text)

# Subnets públicas recebem IP público automaticamente
aws ec2 modify-subnet-attribute --subnet-id $PUB_A --map-public-ip-on-launch
aws ec2 modify-subnet-attribute --subnet-id $PUB_B --map-public-ip-on-launch

# Tags
for id in $PUB_A $PUB_B $PRIV_A $PRIV_B; do
  aws ec2 create-tags --resources $id --tags Key=Name,Value=$APP-subnet
done

echo "PUB_A=$PUB_A  PUB_B=$PUB_B  PRIV_A=$PRIV_A  PRIV_B=$PRIV_B"  # ANOTE!
Passo 3 — IGW, NAT Gateway e Route Tables
# Internet Gateway: ponto de entrada/saída para a internet
IGW=$(aws ec2 create-internet-gateway --query 'InternetGateway.InternetGatewayId' --output text)
aws ec2 attach-internet-gateway --vpc-id $VPC_ID --internet-gateway-id $IGW
aws ec2 create-tags --resources $IGW --tags Key=Name,Value=$APP-igw

# Elastic IP para o NAT Gateway (IP fixo de saída)
EIP=$(aws ec2 allocate-address --domain vpc --query 'AllocationId' --output text)

# NAT Gateway: permite que instâncias PRIVADAS acessem a internet
# SEMPRE em subnet PÚBLICA — é o ponto de saída das privadas
# Custo: ~$1/hora — delete ao fim do lab! (aws ec2 delete-nat-gateway)
NAT=$(aws ec2 create-nat-gateway \
  --subnet-id $PUB_A \
  --allocation-id $EIP \
  --query 'NatGateway.NatGatewayId' --output text)

echo "Aguardando NAT Gateway ficar disponível (1-2 min)..."
aws ec2 wait nat-gateway-available --nat-gateway-ids $NAT

# Route Table PÚBLICA: 0.0.0.0/0 → IGW
RT_PUB=$(aws ec2 create-route-table --vpc-id $VPC_ID --query 'RouteTable.RouteTableId' --output text)
aws ec2 create-route --route-table-id $RT_PUB --destination-cidr-block 0.0.0.0/0 --gateway-id $IGW
aws ec2 associate-route-table --route-table-id $RT_PUB --subnet-id $PUB_A
aws ec2 associate-route-table --route-table-id $RT_PUB --subnet-id $PUB_B

# Route Table PRIVADA: 0.0.0.0/0 → NAT GW (NÃO direto para IGW)
RT_PRIV=$(aws ec2 create-route-table --vpc-id $VPC_ID --query 'RouteTable.RouteTableId' --output text)
aws ec2 create-route --route-table-id $RT_PRIV --destination-cidr-block 0.0.0.0/0 --nat-gateway-id $NAT
aws ec2 associate-route-table --route-table-id $RT_PRIV --subnet-id $PRIV_A
aws ec2 associate-route-table --route-table-id $RT_PRIV --subnet-id $PRIV_B

echo "Rede criada! VPC=$VPC_ID | NAT=$NAT"
echo "Salve estas variáveis — serão usadas nos próximos passos."
  1. VPC → Your VPCs → Create VPC · CIDR: 10.0.0.0/16 · Name: techstock-vpc · Enable DNS hostnames ✓ · Enable DNS support ✓
  2. VPC → Subnets → Create subnet · Crie 4 subnets: 10.0.1.0/24 (pub-1a), 10.0.2.0/24 (pub-1b), 10.0.10.0/24 (priv-1a), 10.0.11.0/24 (priv-1b)
  3. Selecione cada subnet pública → Actions → Edit subnet settings → Enable auto-assign public IPv4
  4. VPC → Internet Gateways → Create · Name: techstock-igw → Attach to VPC → selecione techstock-vpc
  5. VPC → Elastic IPs → Allocate (para o NAT GW)
  6. VPC → NAT Gateways → Create · Subnet: pub-1a (pública!) · Elastic IP: o que acabou de criar · Aguarde 'Available'
  7. VPC → Route Tables → Create route table pública → Add route: 0.0.0.0/0 → Internet Gateway → associe às subnets públicas
  8. Crie route table privada → Add route: 0.0.0.0/0 → NAT Gateway → associe às subnets privadas
2
Security Groups
SG-ALB · SG-Backend · SG-DB · SG-Monitoring
▼
🔒
Por que este passo aqui?
Os Security Groups precisam ser criados antes de qualquer EC2, RDS ou ALB, porque esses recursos referenciam os SGs no momento da criação — você não pode criar um EC2 sem informar qual SG ele usará. Criamos todos os SGs agora, de uma vez, para que possamos configurar as regras de acesso cruzado (ex: SG-Backend aceita do SG-ALB) sem retrabalho. A ordem importa: SG-ALB → SG-Backend → SG-DB → SG-Monitoring.
📥 Precisa: VPC ID (passo 1)📤 Fornece: SG_ALB, SG_BACKEND, SG_DB, SG_MONITORING — usados em todos os recursos dos próximos passos
Criar todos os Security Groups
# ── SG do ALB: único ponto de entrada público ──────────────────────────────
SG_ALB=$(aws ec2 create-security-group \
  --group-name $APP-sg-alb \
  --description "TechStock ALB - acesso publico HTTP" \
  --vpc-id $VPC_ID \
  --query 'GroupId' --output text)

aws ec2 authorize-security-group-ingress \
  --group-id $SG_ALB \
  --protocol tcp --port 80 --cidr 0.0.0.0/0

# ── SG do Backend: aceita APENAS do ALB e do Monitoring ─────────────────────
# Usamos SourceSecurityGroupId (não CIDR) — mais seguro e dinâmico
SG_BACKEND=$(aws ec2 create-security-group \
  --group-name $APP-sg-backend \
  --description "TechStock Backend Node.js" \
  --vpc-id $VPC_ID \
  --query 'GroupId' --output text)

# Porta 3000: aceita do ALB (API)
aws ec2 authorize-security-group-ingress \
  --group-id $SG_BACKEND --protocol tcp --port 3000 \
  --source-group $SG_ALB

# Porta 3000 e 9100: aceita do Monitoring (Prometheus raspa métricas)
SG_MONITORING=$(aws ec2 create-security-group \
  --group-name $APP-sg-monitoring \
  --description "TechStock Monitoring Prometheus+Grafana" \
  --vpc-id $VPC_ID \
  --query 'GroupId' --output text)

aws ec2 authorize-security-group-ingress \
  --group-id $SG_BACKEND --protocol tcp --port 3000 \
  --source-group $SG_MONITORING
aws ec2 authorize-security-group-ingress \
  --group-id $SG_BACKEND --protocol tcp --port 9100 \
  --source-group $SG_MONITORING

# ── SG do Banco: aceita APENAS do Backend na porta PostgreSQL ──────────────
SG_DB=$(aws ec2 create-security-group \
  --group-name $APP-sg-db \
  --description "TechStock RDS PostgreSQL" \
  --vpc-id $VPC_ID \
  --query 'GroupId' --output text)

aws ec2 authorize-security-group-ingress \
  --group-id $SG_DB --protocol tcp --port 5432 \
  --source-group $SG_BACKEND

# ── SG do Monitoring: aceita Grafana e Prometheus do ALB ───────────────────
aws ec2 authorize-security-group-ingress \
  --group-id $SG_MONITORING --protocol tcp --port 3000 \
  --source-group $SG_ALB
aws ec2 authorize-security-group-ingress \
  --group-id $SG_MONITORING --protocol tcp --port 9090 \
  --source-group $SG_ALB

# ── SG do Frontend: aceita HTTP do ALB (Nginx serve os estáticos) ──────────
SG_FRONTEND=$(aws ec2 create-security-group \
  --group-name $APP-sg-frontend \
  --description "TechStock Frontend Nginx" \
  --vpc-id $VPC_ID \
  --query 'GroupId' --output text)

# Porta 80: aceita do ALB (Nginx serve os arquivos estáticos)
aws ec2 authorize-security-group-ingress \
  --group-id $SG_FRONTEND --protocol tcp --port 80 \
  --source-group $SG_ALB

echo "SG_ALB=$SG_ALB"
echo "SG_BACKEND=$SG_BACKEND"
echo "SG_DB=$SG_DB"
echo "SG_MONITORING=$SG_MONITORING"
echo "SG_FRONTEND=$SG_FRONTEND"
echo "Salve estas variáveis!"
  1. EC2 → Security Groups → Create security group
  2. Crie 5 SGs na techstock-vpc: sg-alb, sg-backend, sg-db, sg-monitoring, sg-frontend
  3. sg-alb → Inbound: TCP 80 source 0.0.0.0/0
  4. sg-backend → Inbound: TCP 3000 source sg-alb · TCP 3000 source sg-monitoring · TCP 9100 source sg-monitoring
  5. sg-db → Inbound: TCP 5432 source sg-backend
  6. sg-monitoring → Inbound: TCP 3000 source sg-alb · TCP 9090 source sg-alb · (Nginx na porta 80 se usar proxy)
  7. sg-frontend → Inbound: TCP 80 source sg-alb (Nginx serve os arquivos estáticos do TechStock)

Banco de Dados — RDS PostgreSQL
db.t3.micro · gp2 · Subnet privada · SSL obrigatório
▼
🗄️
Por que este passo aqui?
O RDS é criado antes do Backend porque o backend precisa do endpoint do RDS (ex: techstock-db.abc.us-east-1.rds.amazonaws.com) para configurar o arquivo .env. Se criássemos o backend primeiro, teríamos que voltar e reconfigurar o .env — exatamente o vai-e-vem que queremos evitar. O RDS demora ~5-10 minutos para ficar disponível, então criamos agora e configuramos o backend enquanto o banco inicializa.
📥 Precisa: VPC ID (passo 1)📥 Precisa: SG_DB (passo 2)📥 Precisa: PRIV_A e PRIV_B (passo 1)📤 Fornece: RDS_ENDPOINT — essencial para o .env do Backend (passo 4)
Criar Subnet Group e instância RDS PostgreSQL
# Subnet Group: informa em quais subnets o RDS pode ser colocado
# Mínimo 2 subnets em 2 AZs diferentes (mesmo que o RDS fique em apenas 1)
aws rds create-db-subnet-group \
  --db-subnet-group-name $APP-db-subnet-group \
  --db-subnet-group-description "TechStock DB Subnets" \
  --subnet-ids $PRIV_A $PRIV_B

# ── Cria o RDS PostgreSQL ────────────────────────────────────────────────────
# Limitações OBRIGATÓRIAS do Learner Lab:
#   --no-monitoring-interval ou --monitoring-interval 0 → sem Enhanced Monitoring
#   --no-enable-performance-insights → não suportado
#   --storage-type gp2 → gp3 e PIOPS não suportados

#   --no-multi-az → suportado mas caro para laboratório

aws rds create-db-instance \
  --db-instance-identifier $APP-db \
  --db-instance-class db.t3.micro \
  --engine postgres \
  --engine-version "15" \
  --master-username techstock_user \
  --master-user-password "SenhaForte@2024!" \
  --db-name techstock \
  --allocated-storage 20 \
  --storage-type gp2 \
  --vpc-security-group-ids $SG_DB \
  --db-subnet-group-name $APP-db-subnet-group \
  --no-publicly-accessible \
  --no-multi-az \
  --monitoring-interval 0 \
  --no-enable-performance-insights \
  --backup-retention-period 0

echo "RDS criando... (5-10 min). Continue com os próximos passos enquanto aguarda."
echo "Para verificar: aws rds describe-db-instances --db-instance-identifier $APP-db --query 'DBInstances[0].DBInstanceStatus'"

EC2 Backend — Node.js
t2.small · subnet privada · SSM · systemd · .env · schema.sql
▼
🖥️
Por que este passo aqui?
O Backend é configurado logo após o banco porque precisamos do RDS_ENDPOINT para preencher o .env corretamente (DB_HOST). A ordem garante que não precisamos voltar para ajustar variáveis. O Backend precisa estar rodando antes de criar o ALB porque o ALB registra a instância em seu Target Group e faz health checks — se o backend não estiver de pé quando o ALB for criado, os checks falharão.
📥 Precisa: RDS_ENDPOINT (passo 3)📥 Precisa: SG_BACKEND (passo 2)📥 Precisa: PRIV_A (passo 1)📥 Precisa: NAT Gateway ativo (passo 1) — para npm install📤 Fornece: EC2_BACKEND ID e IP privado📤 Fornece: API rodando em :3000📤 Fornece: /metrics para o Prometheus (passo 9)
Lançar EC2 Backend
# Busca a AMI mais recente do Amazon Linux 2023
AMI_ID=$(aws ec2 describe-images --owners amazon \

5
ALB + Target Groups + Listener Rules
internet-facing · roteamento por path · health checks
▼
⚖️
Por que este passo aqui?
O ALB é criado após o Backend estar rodando porque o Target Group registra a instância do backend e inicia os health checks imediatamente. Se o backend não estivesse de pé, o ALB ficaria com targets 'Unhealthy'. O ALB é criado antes do Frontend porque o DNS do ALB (techstock-lb-XXX.elb.amazonaws.com) precisa ir para o config.js do frontend — sem o DNS do ALB, o frontend não sabe para onde apontar as chamadas de API.
📥 Precisa: EC2_BACKEND rodando (passo 4)📥 Precisa: SG_ALB e SG_BACKEND (passo 2)📥 Precisa: PUB_A e PUB_B (passo 1)📥 Precisa: VPC_ID (passo 1)📤 Fornece: ALB_DNS — vai para o config.js do frontend (passo 8) e CORS_ORIGIN do backend (passo 7)📤 Fornece: ALB_ARN e TG_BACKEND_ARN
Criar Target Groups e ALB com regras de roteamento
# ── Target Group Backend ────────────────────────────────────────────────────
TG_BACKEND=$(aws elbv2 create-target-group \
  --name $APP-tg-backend \
  --protocol HTTP --port 3000 \
  --vpc-id $VPC_ID \
  --health-check-path /api/health \
  --health-check-interval-seconds 30 \
  --healthy-threshold-count 2 \
  --unhealthy-threshold-count 3 \
  --query 'TargetGroups[0].TargetGroupArn' --output text)

# Registra a instância backend no Target Group
aws elbv2 register-targets \
  --target-group-arn $TG_BACKEND \
  --targets Id=$EC2_BACKEND,Port=3000

# ── Target Group Monitoring ─────────────────────────────────────────────────
TG_MONITORING=$(aws elbv2 create-target-group \
  --name $APP-tg-monitoring \
  --protocol HTTP --port 3000 \
  --vpc-id $VPC_ID \
  --health-check-path /grafana/api/health \
  --health-check-interval-seconds 60 \
  --query 'TargetGroups[0].TargetGroupArn' --output text)

# ── ALB: internet-facing = IP público, aceita tráfego da internet ───────────
ALB_ARN=$(aws elbv2 create-load-balancer \
  --name $APP-alb \
  --type application \
  --scheme internet-facing \
  --security-groups $SG_ALB \
  --subnets $PUB_A $PUB_B \
  --query 'LoadBalancers[0].LoadBalancerArn' --output text)

ALB_DNS=$(aws elbv2 describe-load-balancers \
  --load-balancer-arns $ALB_ARN \
  --query 'LoadBalancers[0].DNSName' --output text)

echo "ALB_DNS=$ALB_DNS"  # ANOTE! Vai para config.js e CORS_ORIGIN
